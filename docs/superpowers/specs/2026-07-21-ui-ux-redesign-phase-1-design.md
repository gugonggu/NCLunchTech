# 전체 UI·UX 리디자인 1차 설계

## 목표와 단계

앤시점심기술의 기존 기능, URL, 서버 동작과 비즈니스 규칙을 유지하면서 사용자가 점심시간 전에 1~2분 안에 식당과 동행 여부를 결정할 수 있는 일관된 모바일 중심 인터페이스로 개선한다. 최종 범위는 전체 사용자·관리자 화면이지만, 검증 가능한 세 구현 사이클로 나눈다.

1. 디자인 시스템, 전역 레이아웃, 로그인·회원가입, 홈, 즉시 추천
2. 식당 목록·상세·지도, 함께 먹기, 투표, N빵, 알림
3. 프로필·도감·리더보드, 관리자, 전체 반응형·접근성 마감

이 문서와 후속 구현 계획의 범위는 첫 번째 사이클이다. 각 사이클은 관련 테스트와 브라우저 검토를 마친 뒤 다음 사이클로 진행한다.

## 현재 UI의 핵심 문제

- 전역 디자인 토큰이 브랜드 색상 세 개뿐이고 버튼, 입력, 카드, 배지, 상태 표현이 화면마다 직접 작성돼 있다.
- 루트 레이아웃이 모든 화면에서 `max-w-md`로 고정되어 데스크톱 공간과 지도 화면을 활용하지 못한다.
- 텍스트 기호를 아이콘으로 사용한 4개 하단 메뉴가 약속과 알림 같은 핵심 흐름을 충분히 반영하지 못한다.
- 홈은 추천, 결정, 방문 확인, 약속, 투표, 보조 링크가 동일한 카드·버튼 언어로 나열되어 현재 가장 중요한 행동이 약하다.
- 홈, 추천, 식당 상세, 약속 상세이 각각 280~508줄의 큰 페이지로 구성되어 표시 책임과 상태 분기가 뒤섞여 있다.
- 로그인 폼은 placeholder에 의존하고 서비스 가치 설명, 오류 연결, 명시적인 입력 계층이 부족하다.
- 로딩·빈 상태·오류·성공·제출 중 상태에 공통 표현과 접근성 규칙이 없다.

## 채택한 접근

필요한 토큰과 공통 컴포넌트를 먼저 만들고, 전역 레이아웃에서 로그인, 홈, 추천으로 이어지는 수직 슬라이스를 완성하면서 시스템을 확장한다. 모든 공통 요소를 선제적으로 추상화하지 않으며, 첫 사이클에서 실제로 두 번 이상 사용하는 표현만 공통화한다.

페이지의 데이터 조회, Server Action, API 호출과 상태 판정 결과는 유지한다. 기존 페이지는 데이터 조합과 액션 연결을 담당하고, 표시 책임은 작고 명확한 컴포넌트로 분리한다.

## 변경 경계

첫 사이클에서 변경 가능한 범위는 다음과 같다.

- `src/app/globals.css`의 디자인 토큰과 전역 기본 스타일
- 루트 레이아웃과 전역 내비게이션
- 로그인, 회원가입, 비로그인 홈, 인증된 홈, 즉시 추천 화면
- 첫 사이클에 필요한 공통 UI·레이아웃·점심 도메인 표시 컴포넌트
- 표시 구조 변경에 필요한 UI 테스트와 기존 화면 테스트 보완

다음 항목은 변경하지 않는다.

- DB 스키마, Supabase 테이블과 RLS 정책
- 인증 방식, 세션 정책, 추천 알고리즘과 제외 쿠키
- 방문·약속·투표·N빵·제보·알림의 비즈니스 규칙
- API 응답 구조, Server Action 입력 계약, URL과 쿼리 파라미터
- 정상 동작하는 서버 조회와 저장 흐름

시각 개선 때문에 동작 변경이 불가피한 경우 구현하지 않고 별도 승인을 요청한다. 첫 사이클에는 새 UI 또는 애니메이션 라이브러리를 추가하지 않는다.

## 디자인 시스템

### 색상

- Primary: `#F47C20`
- Primary hover: `#D9600B`
- Primary soft: `#FFF0E2`
- Background: `#FBF8F4`
- Surface: `#FFFFFF`
- Surface muted: `#F5F1EC`
- Text primary: `#292521`
- Text secondary: `#6F6861`
- Border: `#E8E1D9`
- 성공·영업 중: 녹색 계열
- 경고·혼잡: 황갈색 계열
- 오류·영업 종료: 적색 계열
- 정보: 청색 계열

오렌지는 주요 CTA, 선택 상태, 오늘의 핵심 상태와 활성 내비게이션에만 사용한다. 상태는 색상과 함께 아이콘 또는 텍스트 라벨을 제공한다.

### 타이포그래피와 간격

한국어 시스템 폰트 스택을 사용한다. 화면 제목, 섹션 제목, 카드 제목, 본문, 보조 설명, 메타 정보, 상태 라벨의 크기와 굵기를 토큰으로 구분한다. 식당명, 약속 시간과 금액은 주변 정보보다 한 단계 강하게 표시한다.

간격은 4px 단위를 사용한다. 페이지 좌우 여백은 모바일 20px, 데스크톱 32px를 기본으로 하고 화면 목적에 맞는 최대 너비 안에서 적용한다.

### 형태와 상호작용

- 기본 버튼 높이 48px, 압축 버튼 높이 40px
- 입력 필드 최소 높이 48px
- 주요 카드 radius 20px, 버튼과 입력 radius 14px
- 카드 구분에는 얇은 경계, 배경 차이, 약한 그림자 중 필요한 수단만 사용
- 모든 클릭 영역은 최소 44px를 확보
- 키보드 포커스 링을 명확히 표시
- hover, press와 콘텐츠 전환은 150~220ms 범위의 짧은 효과만 사용
- `prefers-reduced-motion`에서는 불필요한 전환을 제거

아이콘은 공통 내부 SVG 컴포넌트로 제공한다. 이모지나 텍스트 기호를 내비게이션 아이콘으로 사용하지 않는다.

## 컴포넌트 구조

`src/components/ui`에는 첫 사이클에서 반복되는 Button, Card, Badge, Input, EmptyState, FeedbackMessage, Skeleton만 둔다. 각 컴포넌트는 제한된 variant를 가지며 임의 색상이나 크기를 호출부에서 반복하지 않는다.

`src/components/layout`에는 다음 요소를 둔다.

- 앱 셸과 페이지 최대 너비를 관리하는 컨테이너
- 모바일 하단 내비게이션
- 데스크톱 브랜드 헤더와 가로 내비게이션
- 인증 화면 전용 셸

`src/components/lunch`에는 추천 식당 카드, 오늘 상태 패널처럼 점심 도메인 의미를 가진 표시 컴포넌트를 둔다. 데이터 조회나 Server Action은 이 계층으로 옮기지 않는다.

## 전역 레이아웃과 내비게이션

모바일 하단 메뉴는 `홈 / 식당 / 함께 먹기 / 알림 / 내 정보`로 구성한다. 도감과 리더보드는 내 정보에서 접근한다. 활성 메뉴는 아이콘, 텍스트 색상과 부드러운 배경으로 함께 구분하고 `aria-current="page"`를 유지한다. Safe Area와 내비게이션 높이만큼 콘텐츠 여백을 확보한다.

데스크톱은 상단 브랜드 헤더와 가로 내비게이션을 사용한다. 일반 콘텐츠의 최대 너비는 약 1120px로 두고, 폼과 읽기 중심 화면은 640~760px로 제한한다. 홈은 필요할 때 주요 상태와 보조 정보를 2열로 배치할 수 있다. 지도는 후속 사이클에서 목록과 지도를 나란히 표시할 수 있는 넓은 전용 레이아웃을 사용한다.

로그인과 회원가입에는 내비게이션 없는 인증 셸을 사용한다. 모바일에서는 한 열, 넓은 화면에서는 간단한 브랜드 소개와 폼을 분리하되 불필요한 랜딩 콘텐츠는 추가하지 않는다.

## 화면 설계

### 로그인과 회원가입

- 서비스 이름과 “1~2분 안에 오늘 점심 결정”이라는 핵심 가치를 짧게 표시한다.
- 모든 입력에 보이는 label을 연결하고 placeholder는 예시 또는 보조 정보로만 사용한다.
- PIN 입력, 오류와 제출 중 상태를 관련 필드와 버튼 가까이에 표시한다.
- 모바일 키보드가 표시돼도 입력과 제출 버튼에 접근할 수 있게 수직 여백을 조절한다.
- 로그인 후 기존 `returnTo` 복귀 흐름과 회원가입 입력 계약을 유지한다.
- 비로그인 홈은 로그인과 회원가입으로 자연스럽게 연결되는 간결한 진입 화면으로 유지한다.

### 인증된 홈

홈은 현재 상태에 따라 하나의 주인공 영역만 선택한다.

1. 방문 확인 또는 응답이 필요한 초대·투표
2. 오늘 결정한 식당 또는 참여 중인 약속
3. 결정 전 오늘의 추천 시작
4. 점심 이후 리뷰와 메뉴 기록

더 높은 우선순위 상태가 있으면 해당 상태를 상단 Hero 영역에 표시하고, 나머지는 “오늘 일정”과 보조 섹션으로 낮춘다. 알림은 상단 헤더에서 접근하고, 식당 찾기·도감 같은 보조 기능은 작은 바로가기로 배치한다. 로그아웃은 홈에서 제거하고 내 정보에서 유지한다.

서버의 기존 방문, 약속, 투표, 알림 조회와 상태 판정은 유지한다. 화면 우선순위만 별도 순수 함수 또는 명시적 분기로 표현하고 테스트로 고정한다.

### 즉시 추천

메인 추천은 사진 중심의 대표 카드로 표시한다. 식당명, 카테고리, 거리, 가격대, 영업 상태, 추천 이유와 최근 상태를 우선순위에 따라 배치한다. 사진이 없으면 카테고리 기반의 절제된 fallback을 사용한다.

`여기로 결정`만 Primary CTA로 사용한다. `다시 추천`은 Secondary, `상세 보기`는 낮은 강조의 링크로 구분한다. 대안 식당은 더 작은 카드로 표시하고 메인 추천과 경쟁하지 않게 한다.

추천 조건은 모바일에서 Bottom Sheet, 데스크톱에서 보조 패널로 제공한다. 짧은 선택에만 Bottom Sheet를 사용하며 닫기, Escape, 포커스 복귀, 배경 스크롤 방지와 제출 중 상태를 처리한다. 추천 지도는 결과 이해를 돕는 보조 요소로 유지하고 기존 Kakao 지도 로딩과 마커 동작을 바꾸지 않는다.

## 상태와 오류 처리

- 로딩: 최종 콘텐츠와 유사한 크기의 Skeleton을 사용해 레이아웃 이동을 줄인다.
- 빈 상태: 현재 비어 있는 이유, 짧은 설명과 가능한 다음 행동을 제공한다.
- 오류: Supabase 원문 대신 이해 가능한 메시지와 가능한 경우 재시도 동작을 제공한다.
- 성공: 짧은 인라인 피드백 또는 기존 리다이렉트 결과 메시지를 사용한다.
- 제출 중: 버튼을 비활성화하고 처리 문구를 표시해 중복 요청을 막는다.

긴 식당명, 닉네임과 상태 문구는 고정 폭을 가정하지 않는다. 중요 제목은 자연스럽게 줄바꿈하고, 메타 정보는 필요한 경우 제한된 줄 수와 접근 가능한 전체 텍스트를 제공한다.

## 접근성

- label과 입력을 명시적으로 연결하고 오류 메시지를 `aria-describedby`로 연결한다.
- 내비게이션에 의미 있는 SVG와 텍스트 라벨을 함께 제공한다.
- 아이콘 전용 버튼에는 `aria-label`을 제공한다.
- 상태를 색상만으로 표현하지 않는다.
- 문서 구조에 맞는 heading 순서와 landmark를 사용한다.
- 포커스 표시를 유지하고 키보드로 모든 주요 행동에 접근할 수 있게 한다.
- 모바일 Bottom Sheet는 열릴 때 포커스를 이동하고 닫힐 때 트리거로 복귀한다.
- 이미지에는 내용에 맞는 대체 텍스트를 제공한다.

## 검증 기준

첫 사이클에서 다음을 확인한다.

- 기존 단위 테스트와 페이지 테스트가 유지된다.
- 공통 UI의 variant, 비활성 상태와 접근 가능한 이름을 테스트한다.
- 홈의 상태 우선순위와 기존 링크·Server Action 연결을 테스트한다.
- 추천의 Primary·Secondary 행동과 기존 검색 파라미터·결정 액션 연결을 테스트한다.
- `360`, `390`, `430`, `768`, `1280`, `1440px`에서 가로 스크롤, 콘텐츠 잘림, 내비게이션 겹침과 긴 텍스트를 확인한다.
- 키보드 탐색, 포커스 표시, label, `aria-current`와 reduced motion을 확인한다.
- 로그인, 비로그인 홈, 인증된 홈과 추천을 실제 브라우저에서 모바일·데스크톱으로 검토한다.
- 저장소에 존재하는 `npm run lint`, `npm run typecheck`, `npm run test`, `npm run build`를 실행한다.

인증된 브라우저 상태가 없어 실제 데이터 화면을 확인할 수 없는 경우 임의로 운영 데이터를 만들지 않는다. 테스트 환경 또는 사용자가 제공한 안전한 테스트 계정이 필요하다는 점을 결과에 명시한다.

## 완료 조건

- 첫 사이클 화면이 같은 제품으로 보이는 색상, 타이포그래피, 간격과 상태 언어를 사용한다.
- 모바일에서 현재 가장 중요한 행동이 첫 화면에서 명확하다.
- 데스크톱에서 모바일 폭을 단순히 가운데 늘여 놓은 형태가 아니다.
- 로그인 폼과 내비게이션의 접근성이 기존보다 개선된다.
- 추천 결과에서 `여기로 결정`과 나머지 행동의 우선순위가 분명하다.
