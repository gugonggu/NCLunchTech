# 점심 룰렛 설계

## 목표

기본 추천을 바꾸지 않고, 추천 화면에서 사용자가 설정한 조건에 맞는 식당 후보를 시각적인 룰렛으로 고르는 별도 재미 모드를 제공한다.

## 범위

- `/recommend`의 조건 패널에 `룰렛 모드` 진입 버튼을 추가한다.
- `/roulette`는 추천과 동일한 조건 파라미터를 검증하고, 동일한 후보 필터링·가중치 규칙을 사용한다.
- 후보 전체를 룰렛에 사용하되, 최종 결과는 가중 무작위 선택으로 결정한다. 최근 방문·혼잡 감점과 선호 가점은 기존 추천 엔진과 동일하다.
- 결과 화면에는 `다시 돌리기`, `여기로 결정`, `같이 먹기`, `투표로 넘기기`를 제공한다.
- `다시 돌리기`는 현재 당첨 식당을 브라우저 쿠키의 추천 제외 목록에 추가한 뒤, 같은 조건으로 새 룰렛을 연다. 후보가 모두 제외되면 기존 추천과 같이 제외 목록을 무시하고 다시 후보를 구성한다.
- `여기로 결정`은 기존 개인 방문 결정 액션을 호출한다.
- `같이 먹기`는 당첨 식당을 선택한 약속 생성 화면으로 이동한다.
- `투표로 넘기기`는 당첨 식당이 미리 선택된 식당 투표 생성 화면으로 이동한다. 사용자는 추가 후보를 선택하고 마감 시각을 입력한 뒤 기존 투표 생성 액션을 사용한다.

## 화면과 데이터 흐름

1. 사용자는 `/recommend`에서 기존 필터를 설정하고 `룰렛 모드`를 누른다.
2. `/roulette` 서버 페이지는 `/recommend`와 같은 쿼리 파라미터, 후보 조회, 상태/리뷰/방문 데이터, `filterCandidates`를 사용해 후보를 만든다.
3. 클라이언트 룰렛 컴포넌트는 후보 이름을 회전 중 표시하고, 1.2초 뒤 서버에서 전달한 가중치 결과를 표시한다. 애니메이션은 결과를 꾸미기 위한 것이며 결과 선택 권한은 서버에 있다.
4. 당첨 결과의 네 행동은 기존 방문 결정, 약속 생성, 투표 생성 경로로 이어진다.

## 오류와 경계

- 회사 좌표가 없거나 조건에 맞는 후보가 없으면 추천 화면과 같은 안내와 `/recommend` 복귀 링크를 표시한다.
- 로그인하지 않은 사용자는 룰렛 조회는 가능하지만 `다시 돌리기`와 방문 결정은 기존 서버 액션의 인증 규칙을 따른다. 약속·투표 화면도 기존 로그인 보호를 유지한다.
- 룰렛은 새 DB 테이블이나 외부 API를 추가하지 않는다. 결과를 저장하지 않으며, 방문/약속/투표는 사용자가 명시적으로 선택할 때만 생성한다.

## 검증

- 순수 룰렛 선택 로직은 후보 누락, 제외 목록, 결정적 난수 주입을 단위 테스트한다.
- 추천 화면 테스트는 조건을 보존한 룰렛 링크를 확인한다.
- 룰렛 컴포넌트 테스트는 회전 완료 후 서버가 제공한 결과와 네 결과 행동 링크/폼이 표시되는지 확인한다.
- 전체 `lint`, `typecheck`, `test`, `build`를 실행한다.
