# 실제 먹은 메뉴·가격 기록 설계

## 목표와 범위

완료된 개인 방문 또는 동료 약속마다 대표 메뉴 한 개와 실제 지불 가격을 기록한다. 등록 메뉴 선택과 직접 입력을 모두 지원하며, 식당 메뉴가 바뀌거나 삭제돼도 과거 기록은 메뉴명·가격 스냅샷으로 유지한다. 기존 식당별 리뷰 작성·수정 흐름은 유지하고 같은 화면에서 접근할 수 있게 한다.

사진, 영수증, 여러 메뉴, 메뉴 반응, 인기 순위, 가격 통계는 포함하지 않는다.

## 데이터 구조

`0012_meal_records.sql`에서 `meal_records` 테이블 하나를 추가한다.

- `id`, `employee_id`, `restaurant_id`
- `visit_id` 또는 `appointment_id` 중 정확히 하나
- 선택한 등록 메뉴를 가리키는 nullable `menu_item_id`
- 변경되지 않는 `menu_name_snapshot`, `paid_price`
- `created_at`, `updated_at`

개인 방문은 `visit_id`를 유일하게 하고, 약속은 `(employee_id, appointment_id)`를 유일하게 한다. 따라서 같은 완료 기록을 다시 저장하면 새 행을 만들지 않고 기존 행을 수정한다. `menu_item_id` 삭제 시에는 `null`로 바꾸되 스냅샷은 보존한다. 메뉴명은 공백 제거 후 1~100자, 가격은 0~10,000,000원의 정수로 DB와 서버에서 검증한다.

## 서버 검증과 저장 흐름

폼은 `restaurantId`와 함께 `visitId` 또는 `appointmentId` 하나를 전달한다. 서버는 로그인 직원을 기준으로 출처를 다시 조회하고 다음을 검증한다.

- 개인 방문: 요청 직원 소유, 요청 식당 소속, 상태 `completed`
- 약속 방장: 요청 직원이 방장, 요청 식당 소속, `host_attendance_status = completed`
- 약속 참여자: 본인 참여 행이 `completed`, 약속이 요청 식당 소속
- 등록 메뉴: 요청 식당 소속이며 서버에서 조회한 현재 이름을 스냅샷으로 사용
- 직접 입력: 등록 메뉴 ID 없이 유효한 직접 메뉴명을 사용
- 등록 메뉴와 직접 입력은 정확히 하나만 선택

검증이 끝나면 출처별 유일 키로 upsert한다. 클라이언트가 보낸 직원 ID, 식당 소유 관계, 등록 메뉴 이름은 신뢰하지 않는다. 다른 직원·식당 데이터나 완료되지 않은 기록은 안전한 상태 코드로 거부한다.

## 화면과 조회

기존 `/reviews/new` 화면에 독립된 “먹은 메뉴 기록” 폼을 추가한다. 완료된 개인 방문 링크는 `visitId`, 완료된 약속 링크는 `appointmentId`를 포함한다. 출처 ID가 없는 식당 상세의 기존 링크는 리뷰만 작성·수정할 수 있다.

기록 폼에는 해당 식당의 등록 메뉴 선택, 직접 메뉴명, 실제 지불 가격을 표시한다. 기존 기록이 있으면 값을 채워 수정할 수 있게 한다. 저장 후 같은 화면으로 돌아와 성공 또는 검증 오류를 표시한다. 리뷰 저장은 기존 식당별 1개 upsert 정책을 그대로 유지한다.

홈의 오늘 완료 방문과 약속 상세에는 해당 기록을 표시한다. 도감에서는 식당별 가장 최근에 생성된 본인 기록의 메뉴명과 가격을 표시한다. 기록이 없어도 기존 화면은 그대로 동작한다.

## 테스트와 적용

서버 입력 검증은 등록/직접 입력 선택, 메뉴명 길이, 정수 가격 범위를 단위 테스트한다. 출처 판정은 개인 방문, 약속 방장, 약속 참여자의 완료 여부와 직원·식당 불일치를 테스트한다. 마이그레이션에는 출처 XOR, 유일 키, 스냅샷·가격 제약을 확인하는 정적 테스트를 둔다. 화면에는 기록 유무와 출처 링크 파라미터를 확인하는 최소 테스트만 추가한다.

마이그레이션 파일은 로컬에서 작성할 수 있지만 원격 DB 적용은 별도 승인을 받은 뒤 실행한다. 기존 데이터는 수정하거나 보정하지 않는다.
