# 13-C 관리자 원자성·방문 확인 시간 설계

## 목표

13-B에서 실패 감지만 추가했던 관리자 복합 쓰기를 PostgreSQL 트랜잭션으로 원자화한다. 개인 방문과 동료 약속의 방문 확인 가능 시점을 사용자 결정에 맞게 분리하고, 잘못된 달력 날짜와 미래 약속 조기 확인을 서버에서 차단한다.

## 확정 정책

- 개인 방문은 `planned` 상태가 생성된 직후부터 `다녀왔어요` 또는 `가지 않았어요`로 확인할 수 있다.
- 개인 방문에는 별도 예정 시각 입력이나 `scheduled_at` 컬럼을 추가하지 않는다.
- 동료 약속은 `scheduled_at`이 현재 시각과 같거나 지난 뒤부터 방문 확인할 수 있다.
- 방장과 참여자 모두 미래 약속을 방문 완료 또는 불참으로 확인할 수 없다.
- 약속 생성과 변경은 현재보다 미래인 시각만 허용한다.
- `2026-02-31T12:30`처럼 형식은 맞지만 존재하지 않는 Asia/Seoul 달력 시각은 거부한다.

## 관리자 트랜잭션 구조

`0011_admin_atomic_operations.sql`에 업무별 `security definer` 함수를 추가한다. 각 함수는 입력 대상 행을 잠그고 데이터 변경과 `admin_logs` 삽입을 같은 트랜잭션에서 수행한다. 함수 실행 중 하나라도 실패하면 PostgreSQL이 전체 작업을 rollback한다.

함수 경계는 다음과 같다.

- 직원 PIN 초기화: PIN 해시·로그인 실패 상태 초기화, 활성 세션 폐기, 감사 로그
- 직원 활성 상태 변경: 활성 상태·비활성 시각 변경, 비활성화 시 활성 세션 폐기, 감사 로그
- 신고 기각: 대기 신고를 해결 상태로 변경, 감사 로그
- 신고된 리뷰 삭제: 대기 신고와 리뷰 잠금, 리뷰 snapshot 확보, 관련 신고와 리뷰 삭제, 감사 로그
- CSV batch 반영: 대기 batch 잠금, 검증 완료 행 전체 반영, batch 완료 표시, 감사 로그

함수는 명시적으로 `search_path`를 비우고 모든 객체를 스키마로 한정한다. `public`, `anon`, `authenticated`의 실행 권한을 제거하고 `service_role`에만 부여한다. 관리자 존재 여부와 대상 상태는 함수 안에서도 확인한다.

## CSV 원자성

Server Action은 13-B의 Zod 스키마로 저장 batch를 다시 검증한 뒤 batch ID와 관리자 ID만 RPC에 전달한다. RPC는 `csv_import_batches` 행을 `for update`로 잠그고 `pending` 상태인지 다시 확인한다.

CSV 반영 함수는 transaction advisory lock으로 여러 batch의 동시 반영을 직렬화한다. 메뉴는 동일 식당·메뉴명을 찾아 수정하거나 없으면 추가하고, 영업시간은 기존 `(restaurant_id, day_of_week)` 고유 제약을 사용해 upsert한다. 유효 행이 없거나 참조 식당이 없거나 저장 JSON이 예상 구조와 다르면 batch 상태와 일부 데이터 모두 변경하지 않는다.

## 방문 확인 시간

기존의 공통 `기준 시각 + 1시간` helper를 제거한다. 개인 방문 화면은 오늘의 `planned` 방문이면 즉시 확인 대상으로 분류한다. 개인 방문 완료·취소 Action은 현재 상태가 `planned`인지 다시 확인한다.

동료 약속 화면과 홈 조회는 `scheduled_at <= now`를 확인 가능 경계로 사용한다. 방문 확인 Action도 약속 상태, 사용자 역할·참여 상태와 함께 동일한 시간 조건을 검사한다. UI를 우회해 Server Action을 직접 호출해도 미래 약속은 변경되지 않고 허용 목록 상태 메시지로 돌아간다.

## 날짜 파싱

`parseSeoulDateTimeLocal`은 정규식 일치 후 월·일·시·분 범위를 확인하고, UTC 변환 결과를 Asia/Seoul 벽시계 문자열로 다시 변환해 원래 입력과 동일한지 검사한다. JavaScript `Date.UTC`의 자동 날짜 이월은 유효한 입력으로 인정하지 않는다.

## Server Action

관리자 Action은 인증과 Zod 입력 검증을 유지하고 복합 쓰기를 RPC 한 번으로 교체한다. RPC의 안정된 결과 코드는 기존 허용 목록 상태로 변환하며 내부 SQL 오류는 사용자에게 노출하지 않는다.

방문·약속 Action은 UUID·권한·현재 상태·시간 경계를 서버에서 검증한다. 경쟁 요청으로 영향 행이 없을 때 성공으로 표시하지 않고 이미 처리됨 또는 조기 확인 상태로 안내한다.

## 마이그레이션 적용 안전 절차

원격 DB 적용 전 다음을 읽기 전용으로 확인한다.

- `0010`까지의 원격 구조와 로컬 기준선 일치 여부
- 메뉴의 동일 식당·메뉴명 중복 현황
- 대기 CSV batch와 저장 JSON 구조
- 대기 신고의 리뷰 FK, 직원과 활성 세션 FK
- 생성할 함수 이름과 기존 객체 충돌 여부

`0011`은 함수와 권한만 추가하며 운영 행을 삭제하거나 일괄 수정하지 않는다. 실제 원격 적용 직전에 전체 SQL, 생성 객체, 권한, 예상 영향을 제시하고 별도 확인을 받는다.

## 테스트

- 개인 방문은 생성 직후 확인 대상으로 분류되는지 테스트한다.
- 동료 약속은 예정 시각 직전에는 확인 불가, 정확히 예정 시각부터 확인 가능인지 테스트한다.
- 미래 약속의 방장·참여자 확인 요청이 거부되는지 테스트한다.
- 유효하지 않은 날짜, 월, 일, 시, 분을 파서가 거부하는지 테스트한다.
- 관리자 Action이 각 업무 RPC를 한 번 호출하고 결과를 안전한 상태 코드로 변환하는지 테스트한다.
- 테스트 DB가 준비된 경우 RPC별 성공·rollback·중복 요청을 통합 테스트한다.
- 최종 파일 상태에서 lint, typecheck, unit test, build를 실행한다.
- 테스트 DB가 없으면 integration과 E2E는 미실행으로 보고하며 통과로 표현하지 않는다.

## 제외

- 약속 생성·참여자 초대·알림 생성의 트랜잭션 리팩터링
- 약속 변경·취소 알림의 트랜잭션 리팩터링
- 개인 방문 시간 선택 UI와 `visits.scheduled_at`
- 새 관리자 기능과 UI 개편
- 초대 코드 변경
- 운영 데이터 삭제 또는 정리
- Git push
