# 13-B 관리자 검증·오류 처리 설계

## 목표

관리자 화면의 모든 외부 입력과 저장된 JSON을 서버에서 검증하고, Supabase 오류와 영향 행 0건을 성공으로 오인하지 않도록 한다. 예상 가능한 실패는 현재 화면의 상태 메시지로 안내하고 내부 오류 세부 정보는 노출하지 않는다.

## 범위

- 관리자 Action 인자의 UUID와 설정값 검증
- 초대 코드 4~64자 및 제어문자 금지, 공지 최대 200자, 기본 반경 허용값 제한
- CSV 최대 2 MiB·5,000행·UTF-8·CSV 확장자 검증과 저장 batch JSON 재검증
- 신고와 리뷰의 실제 관계, 메뉴와 식당의 실제 관계, 변경 이력 snapshot 검증
- 모든 Supabase 오류와 대상 없음 검사, 관리자 로그 insert 오류 전파
- 관리자 화면의 성공·실패 상태 메시지
- 단위 테스트와 전체 품질 검사

## 제외

- `0011` 이상 마이그레이션, RPC, DB 트랜잭션
- CSV·신고·직원 작업의 원자성 보장
- 방문·약속 시간 검증
- 새 관리자 기능

## 구조

`src/lib/admin/validation.ts`가 입력과 저장 JSON의 단일 검증 경계를 담당한다. `src/lib/admin/db-result.ts`는 Supabase 오류와 영향 행 0건을 일관되게 실패로 바꾸며, 각 Action은 검증된 값만 DB에 전달한다. 페이지는 기존 `searchParams.status` 패턴으로 제한된 메시지만 표시한다.

## 오류 처리

입력 오류·대상 없음·이미 처리됨은 안정된 상태 코드로 리다이렉트한다. DB와 감사 로그 오류는 일반화된 서버 오류로 처리하여 내부 메시지를 브라우저에 노출하지 않는다. 13-B는 실패를 감지하지만 여러 쿼리를 rollback하지는 않으며 이 한계는 13-C에서 해결한다.

## 테스트

검증 스키마와 DB 결과 helper는 실패 테스트부터 작성한다. CSV 형식·크기·행 수와 snapshot 변조를 단위 테스트하고, 기존 전체 테스트·lint·typecheck·build를 최종 실행한다.
