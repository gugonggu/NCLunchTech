# 오늘 같이 먹기 상태 공유 설계

## 목표

직원이 정식 약속을 만들기 전에 오늘 점심 상태를 가볍게 공유하고, 점심 동료를 빠르게 찾을 수 있게 한다.

## 범위

- 직원은 서울 날짜 기준 당일 상태를 선택, 변경 또는 해제할 수 있다.
- 상태는 `looking_for_company`, `has_appointment`, `eating_alone`, `away_or_skipping` 네 가지다.
- 로그인한 모든 직원은 상태를 등록한 동료의 닉네임과 상태를 볼 수 있다.
- 홈에서 상태 선택과 상태별 목록을 모두 제공한다.
- 직원당 하루 한 행만 저장한다.

## 제외 범위

- 약속 자동 생성, 자동 초대, 투표 생성, 알림 발송
- 관리자용 개인 상태 통계나 근무 감시 기능
- 미래 날짜 상태 예약 및 과거 상태 수정

## 데이터 모델

`lunch_availabilities` 테이블을 새로 만든다.

- `id uuid`: 기본 키
- `employee_id uuid`: `employees.id` 외래 키, 삭제 시 함께 삭제
- `availability_date date`: Asia/Seoul 기준의 상태 날짜
- `status text`: 네 허용 상태 중 하나
- `created_at`, `updated_at timestamptz`

`(employee_id, availability_date)`에 유니크 제약을 둔다. RLS는 기존 직원 데이터 테이블과 같이 활성화하되 정책을 두지 않아 서비스 롤 서버 코드만 접근한다.

## 서버 동작

상태 변경 Server Action은 현재 세션의 직원을 확인하고, 요청으로 받은 상태가 허용 값인지 검증한다. 날짜와 직원 ID는 클라이언트에서 받지 않고 서버에서 계산한다. 등록 또는 변경은 upsert로 처리한다. 상태 해제는 오늘 행만 삭제한다. 인증되지 않았거나 잘못된 상태는 홈으로 안전하게 복귀시킨다.

조회는 오늘 날짜의 행을 `employees.nickname`과 함께 가져오며, 닉네임 오름차순으로 정렬한다. 홈은 내 상태와 전체 상태 목록을 한 번의 조회 결과로 사용한다.

## 홈 UI

홈의 핵심 추천 카드 아래, 바로가기 위에 `오늘 같이 먹기` 카드를 둔다.

- 아직 등록하지 않은 경우: 네 상태 버튼과 빈 상태 안내를 표시한다.
- 등록한 경우: 현재 내 상태를 표시하고, 네 상태 버튼으로 바로 변경하거나 `상태 해제`로 삭제할 수 있다.
- 상태 목록: 네 상태를 고정 순서로 표시한다. 각 그룹에는 등록한 직원의 닉네임을 나열하고, 비어 있으면 그룹을 숨긴다.
- `같이 먹을 사람을 구해요` 그룹은 인원 수를 함께 표시하고 시각적으로 우선한다.
- 아무도 등록하지 않은 경우: `아직 오늘의 점심 상태를 공유한 동료가 없어요.`를 표시한다.

상태는 당일에만 조회하므로 자정 이후에는 이전 상태가 자동으로 보이지 않는다. 자동 삭제 작업은 필요 없다.

## 오류 및 경계 조건

- 상태 값은 서버에서 화이트리스트 검증한다.
- 유니크 제약과 upsert로 중복 행을 방지한다.
- 상태 조회 실패 시 홈의 다른 기능을 막지 않고 상태 카드에 안내를 표시한다.
- 비활성 직원은 세션 조회가 불가능하므로 등록할 수 없다.

## 검증

- 순수 검증 로직: 허용 상태, 상태 그룹 순서, 당일 상태 선택을 단위 테스트한다.
- 조회/변경 로직: 테스트 가능한 쿼리 입력 및 반환 변환을 단위 테스트한다.
- 홈 UI: 등록 전/후, 전체 빈 상태, 공개된 상태별 닉네임 표시를 컴포넌트 테스트한다.
- 전체 검증: lint, typecheck, 단위 테스트, build를 실행한다.
