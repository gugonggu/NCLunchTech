# 함께 먹기 식당 검색·선택 화면 설계

## 문제

공통 내비게이션의 `함께 먹기` 링크는 `/appointments/new`로 이동한다. 404를 없애기 위해 활성 식당을 전부 조회·렌더링하도록 수정했지만, 데이터가 늘어날수록 응답 시간·메모리·DOM 크기가 함께 증가하고 사용자가 원하는 식당을 찾기 어렵다.

## 목표

`/appointments/new`에서 식당을 검색·필터링해 20개씩 탐색하고, 식당을 선택하면 기존 약속 생성 폼으로 이어지게 한다. 필터·정렬·전체 건수·페이지 분리는 Supabase에서 처리하며 애플리케이션은 전체 식당을 가져오지 않는다.

함께 사용하는 `/restaurants` 지도 화면에서는 검색 영역과 지도 영역의 레이아웃 경계를 분리한다. 바텀시트가 검색 영역 뒤로 들어가지 않게 하고, 사용자가 드래그 핸들뿐 아니라 명시적인 버튼으로도 시트를 화면 아래로 완전히 접고 다시 열 수 있게 한다.

## 사용자 흐름

1. 로그인 사용자가 `/appointments/new`에 접근한다.
2. 기본 조건은 전체 카테고리, 기본 반경, 가까운순, 영업 중 필터 해제, 1페이지다.
3. 사용자는 식당명, 카테고리, 거리, 영업 중 여부, 정렬을 GET 폼으로 변경한다.
4. 검색 결과는 20개씩 표시된다. 이전·다음 이동은 현재 검색 조건을 보존한다.
5. 식당을 선택하면 `/appointments/new?restaurantId=<id>`의 기존 약속 생성 폼으로 이동한다.
6. 약속 생성 폼에서 식당 상세 페이지와 동일한 `카카오맵에서 보기` 링크를 새 탭으로 열 수 있다.

비로그인 사용자는 `/login?returnTo=%2Fappointments%2Fnew`로 이동한다. `restaurantId`가 지정된 비로그인 접근은 해당 ID가 포함된 기존 복귀 경로를 유지한다.

## 검색 조건과 URL

식당 선택 상태는 다음 GET 필드만 사용한다.

- `q`: 식당명 부분 검색. 앞뒤 공백을 제거하고 최대 50자로 제한한다.
- `category`: `RESTAURANT_CATEGORIES` 중 하나 또는 빈 문자열.
- `radius`: `300`, `500`, `800`, `1200`, `2000` 중 하나. 잘못된 값은 회사 기본 반경으로 대체한다.
- `openNow`: `on`이면 서울 현재 시각 기준 영업 중인 식당만 표시한다.
- `sort`: `distance`, `name`, `new` 중 하나. 기본값은 `distance`다.
- `page`: 1 이상의 정수. 잘못된 값은 1로 대체하고 전체 페이지보다 크면 마지막 페이지 결과로 보정한다.

검색 폼은 `page`를 전송하지 않아 조건 변경 시 항상 1페이지부터 시작한다. 페이지 링크는 빈 값과 기본값을 불필요하게 URL에 넣지 않으며 현재 검색 조건을 유지한다.

## 데이터베이스 검색 함수

새 마이그레이션은 `public.search_appointment_restaurants` 함수를 추가한다.

```sql
search_appointment_restaurants(
  p_query text,
  p_category text,
  p_radius_m integer,
  p_open_now boolean,
  p_sort text,
  p_page integer,
  p_page_size integer
)
```

반환 열은 다음과 같다.

```text
id uuid
kakao_place_id text
name text
category text
address text
distance_m integer
is_open_now boolean
total_count bigint
page_number integer
```

함수 계약:

- `restaurants.is_active = true`를 항상 적용한다.
- `app_settings.id = 1`의 `company_lat`, `company_lng`, `default_radius_m`를 사용한다.
- Haversine 거리 계산을 SQL에서 수행하고 지정 반경 밖을 제외한다.
- `restaurant_hours`의 서울 기준 현재 요일·시각으로 `is_open_now`를 계산한다. 기존 애플리케이션과 동일하게 시작 시각 이상, 종료 시각 미만이며 익일 영업은 지원하지 않는다.
- 식당명 부분 검색, 카테고리, 영업 중 조건을 SQL에서 적용한다.
- `distance`, `name`, `created_at desc` 정렬에 식당 ID를 보조 정렬 키로 사용해 페이지 결과를 안정화한다.
- 필터 적용 후 전체 건수를 먼저 계산하고 요청 페이지를 `1..ceil(total_count / page_size)` 범위로 함수 내부에서 보정한다.
- 보정된 `page_number`와 `total_count`를 각 결과 행에 반환하고 `limit 20 offset ((page_number - 1) * 20)`을 적용한다.
- 결과가 0건이면 애플리케이션은 `total_count = 0`, `page_number = 1`로 정규화한다.
- 회사 좌표가 없으면 함수가 `P0001`과 고정 메시지 `company_location_missing`을 반환한다. 애플리케이션은 이 조합만 위치 누락 상태로 변환하고, 다른 RPC 실패는 일반 검색 오류 상태로 처리한다.

함수는 `security invoker`로 만들고 `public`, `anon`, `authenticated`의 실행 권한을 회수한 뒤 `service_role`에만 실행 권한을 부여한다. RLS와 기존 테이블 정책은 변경하지 않는다.

## 검색 인덱스

마이그레이션은 다음 인덱스를 추가한다.

- `pg_trgm` 기반 활성 식당명 부분 검색 인덱스
- 활성 식당 카테고리 인덱스
- 활성 식당 생성일 인덱스
- 활성 식당 위도·경도 복합 인덱스

거리 계산 전에 반경으로부터 위도·경도 bounding box를 계산해 후보 행을 줄이고, 후보에만 정확한 Haversine 필터를 적용한다.

## 애플리케이션 구조

- `src/lib/appointments/restaurant-search.ts`
  - URL 조건 정규화, 페이지 크기 `20`, RPC 호출, 마지막 페이지 보정을 담당한다.
  - 페이지 컴포넌트가 Supabase 쿼리 구현 세부사항을 직접 갖지 않게 한다.
- `src/app/appointments/new/RestaurantPicker.tsx`
  - 검색 폼, 결과 요약, 식당 카드, 이전·다음 페이지 링크를 렌더링한다.
  - 서버 컴포넌트이며 자바스크립트 없이 GET 탐색이 가능하다.
- `src/app/appointments/new/page.tsx`
  - 인증과 `restaurantId` 분기를 유지한다.
  - 식당 미지정 상태에서는 검색 모듈 결과를 `RestaurantPicker`에 전달한다.
  - 식당 지정 상태에서는 기존 약속 생성 폼을 유지하고 `kakao_place_id`를 조회한다.

## UI

- 페이지 제목은 모든 상태에서 `함께 먹기`다.
- 상단 검색 필드와 검색 버튼을 먼저 배치한다.
- 카테고리·거리·영업 중·정렬은 반응형 필터 영역에 배치한다.
- 결과 요약은 `총 N개 · P/T페이지` 형식이다.
- 결과 카드는 식당명, 카테고리, 주소, 거리, `영업 중` 또는 `영업 종료` 상태를 표시한다.
- 카드 전체는 `/appointments/new?restaurantId=<id>` 링크이며 최소 44px 클릭 영역을 유지한다.
- 검색 결과 없음, 회사 위치 누락, RPC 실패 상태를 서로 구분한다.
- 이전·다음 링크는 첫·마지막 페이지에서 각각 비활성 상태로 표시한다.

## 카카오맵 링크

식당 지정 조회에 `kakao_place_id`를 추가한다. 값이 있으면 다음 링크를 약속 생성 폼의 식당 요약 아래에 표시한다.

```text
https://place.map.kakao.com/<kakao_place_id>
```

링크 문구는 `카카오맵에서 보기`이고 `target="_blank"`, `rel="noopener noreferrer"`를 사용한다. 값이 없으면 링크를 렌더링하지 않는다.

## 식당 찾기 지도 바텀시트

### 원인과 레이아웃

현재 `/restaurants`는 검색 헤더와 지도·바텀시트를 동일한 절대 배치 영역에 겹친다. 검색 헤더는 `z-20`, 바텀시트는 `z-10`이므로 시트가 커질수록 검색 헤더 뒤에 가려지고, 상단 콘텐츠와 핸들을 조작할 수 없게 된다.

검색 헤더를 문서 흐름에 있는 축소되지 않는 영역으로 두고, 남은 공간을 별도의 상대 배치 지도 영역으로 만든다. 지도 캔버스와 바텀시트는 이 지도 영역 안에서만 절대 배치한다. 필터 `<details>`가 열려 검색 헤더 높이가 변해도 지도 영역이 그 아래에서 다시 계산되므로 바텀시트가 검색 영역과 겹치지 않는다. 단순히 바텀시트의 `z-index`를 높여 검색창을 덮는 방식은 사용하지 않는다.

### 상태와 조작

`SheetSnap`은 다음 네 상태를 가진다.

- `hidden`: 시트 본문과 핸들을 화면 아래로 완전히 숨긴다.
- `peek`: 제목과 핸들을 확인할 수 있는 최소 높이다.
- `half`: 지도 영역 높이의 약 45%를 사용하며 초기 상태다.
- `full`: 검색 헤더를 제외한 지도 영역 안에서 사용 가능한 높이의 약 85%를 사용한다.

표시된 시트의 위쪽 드래그는 `peek → half → full`, 아래쪽 드래그는 `full → half → peek → hidden` 순서로 인접 상태를 이동한다. 작은 움직임을 탭으로 판정하는 기존 순환 동작은 마지막 단계에서 갑자기 `peek`로 돌아가지 않도록 인접 단계 이동으로 바꾼다. `hidden`에서는 핸들이 없으므로 열기 버튼이나 지도 마커 선택으로만 `half`에 진입한다.

시트 헤더에는 핸들과 별개로 `식당 목록 접기` 버튼을 둔다. 이 버튼은 현재 단계와 관계없이 즉시 `hidden`으로 이동한다. `hidden` 상태에서는 지도 영역 하단에 `식당 목록 열기` 버튼을 표시하며 누르면 `half`로 복원한다. 숨긴 상태에서 지도 마커를 선택해도 선택한 식당을 확인할 수 있도록 `half`로 열린다. 목록에서 식당을 선택할 때는 현재 시트 높이를 유지한다.

접기·열기 버튼의 최소 클릭 영역은 44px이며 `aria-expanded`, 시트 영역을 가리키는 `aria-controls`, 상태를 설명하는 접근 가능한 이름을 제공한다. 핸들은 포인터 조작을 유지하되 버튼 조작만으로 모든 필수 상태 전이가 가능해야 한다.

### 컴포넌트 변경 범위

- `src/app/restaurants/page.tsx`: 검색 헤더와 지도 작업 영역을 일반 흐름으로 분리한다.
- `src/app/restaurants/BottomSheet.tsx`: `hidden` 상태, 명시적 접기 버튼, 인접 단계 상태 전이와 접근성 속성을 담당한다.
- `src/app/restaurants/RestaurantsMapView.tsx`: 완전히 접힌 시트의 열기 버튼과 마커 선택 시 복원 동작을 담당한다.

검색 조건, 서버 데이터 조회, 지도 마커·클러스터링, 식당 상세 링크 동작은 변경하지 않는다.

## 오류와 빈 상태

- 검색 결과 없음: 현재 조건을 완화하도록 안내하고 조건 초기화 링크를 제공한다.
- 회사 위치 누락: `회사 위치 정보가 없어 거리 검색을 사용할 수 없어요`를 표시한다.
- RPC 실패: `식당 검색 결과를 불러오지 못했어요`와 다시 시도 링크를 표시하며 원시 오류를 노출하지 않는다.
- 선택한 식당이 없거나 비활성이면 기존처럼 404를 반환한다.

## 검증

- 정규화 함수가 허용된 검색 조건과 기본값만 RPC에 전달한다.
- RPC SQL 테스트가 활성 필터, 이름·카테고리·반경·영업 중 조건, 세 정렬, 전체 건수, 20개 제한, 페이지 오프셋을 검증한다.
- 페이지 테스트가 RPC 결과만 렌더링하고 전체 식당 `select/order` 조회를 사용하지 않음을 검증한다.
- 조건 변경 시 페이지가 초기화되고 페이지 링크가 조건을 보존하는지 검증한다.
- 빈 결과, 위치 누락, RPC 오류 상태를 검증한다.
- 기존 인증 리디렉션, 지정 식당 404, 약속 생성 폼과 Server Action을 유지하는지 검증한다.
- 카카오맵 링크의 정확한 URL과 보안 속성을 검증한다.
- 실제 브라우저에서 모바일·데스크톱 검색, 필터, 페이지 이동, 식당 선택을 확인한다.
- 바텀시트 컴포넌트 테스트에서 드래그·탭이 인접 상태로 이동하고, 접기 버튼이 어느 상태에서든 `hidden`을 요청하는지 검증한다.
- 지도 화면 컴포넌트 테스트에서 `hidden` 상태의 열기 버튼, 열기 후 `half` 복원, 숨긴 상태에서 마커 선택 시 `half` 복원을 검증한다.
- 데스크톱 브라우저에서 검색 헤더와 지도 영역의 경계, 필터 펼침 후 재배치, `peek`·`half`·`full` 상태의 조작성, 완전 접기와 다시 열기를 확인한다.

## 범위 제외

- 약속 생성 Server Action, 데이터베이스 테이블·RLS, 참여자 처리 방식은 변경하지 않는다.
- 메뉴명·가격·최근 방문 필터는 이번 범위에 포함하지 않는다.
- 무한 스크롤과 클라이언트 상태 관리 라이브러리를 추가하지 않는다.
- `/restaurants`를 데스크톱 사이드 패널 방식으로 전환하거나 지도 검색 조건·데이터 조회를 리팩터링하지 않는다.
