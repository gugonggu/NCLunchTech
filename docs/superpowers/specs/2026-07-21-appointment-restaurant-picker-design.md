# 함께 먹기 식당 선택 화면 설계

## 문제

공통 내비게이션의 `함께 먹기` 링크는 `/appointments/new`로 이동하지만, 현재 페이지는 `restaurantId`가 없으면 `notFound()`를 호출한다. 따라서 정상적인 내비게이션 동작이 404로 끝난다.

## 목표

`/appointments/new`에 직접 접근하면 활성 식당을 선택할 수 있는 화면을 표시하고, 식당을 선택하면 기존 약속 생성 폼으로 이어지게 한다.

## 동작

- 인증 확인은 식당 ID 검사보다 먼저 수행한다.
- 비로그인 사용자는 현재 경로를 보존해 `/login?returnTo=%2Fappointments%2Fnew`로 이동한다.
- 로그인 사용자가 `restaurantId` 없이 접근하면 활성 식당 목록을 이름순으로 조회한다.
- 각 식당은 이름과 카테고리를 보여주며 `/appointments/new?restaurantId=<id>`로 연결한다.
- 활성 식당이 없으면 식당 목록으로 이동할 수 있는 빈 상태를 표시한다.
- `restaurantId`가 있으면 기존 식당 검증, 기본 시각, 참여자·메모 입력, Server Action을 그대로 사용한다.
- 존재하지 않거나 비활성인 `restaurantId`는 기존처럼 404를 반환한다.

## UI

- 제목은 `함께 먹기`로 통일한다.
- 설명은 먼저 식당을 고르도록 안내한다.
- 공유 `Card`, `Badge`, `FeedbackState`, `buttonStyles`를 재사용한다.
- 식당 선택 링크는 최소 44px 클릭 영역과 명확한 포커스 표시를 유지한다.

## 검증

- 식당 ID가 없을 때 404 대신 선택 화면을 렌더링한다.
- 활성 식당 링크가 정확한 쿼리 경로를 가진다.
- 빈 목록 상태와 `/restaurants` 이동 경로를 검증한다.
- 기존 식당 지정 약속 생성 화면과 인증 리디렉션 계약이 유지되는지 검증한다.
- 브라우저에서 `/appointments/new`를 다시 열어 404가 사라졌는지 확인한다.

## 범위 제외

- 약속 생성 Server Action, 데이터베이스 스키마, 참여자 처리 방식은 변경하지 않는다.
- 식당 검색·필터 기능은 이번 수정에 추가하지 않는다.
