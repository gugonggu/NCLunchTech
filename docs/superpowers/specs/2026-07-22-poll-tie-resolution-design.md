# 투표 동점 해결 설계

## 목표

마감된 투표에서 공동 1위가 생기면 방장이 기존 수동 확정 외에 세 가지 자동 기준으로 공동 1위 중 하나를 확정할 수 있게 한다.

## 범위

- 공동 1위가 둘 이상인 closed 투표에서만 방장에게 해결 방식을 보인다.
- 무작위, 회사 기준 최단거리, 최근 덜 방문한 식당 순으로 공동 1위 중 하나를 선택한다.
- 선택 결과는 기존 `decided_option_id`와 `decided_at`에 저장하고 기존 확정 알림을 재사용한다.
- 자동 방식의 마지막 동률은 선택지 position 오름차순으로 결정한다.

## 제외 범위

- 결승 투표 생성, 자동 마감 직후 자동 확정, 투표 결과 변경, 메뉴 투표의 거리 비교
- 개인별 방문 이력 공개 및 관리자 통계

## 동작

식당 투표에만 거리·최근 방문 기준을 제공한다. 무작위 방식은 공동 1위 id 중 하나를 서버에서 선택한다. 최단거리 방식은 회사 좌표와 식당 좌표의 직선 거리를 비교한다. 최근 덜 방문 방식은 회사 전체 완료 방문 수가 작은 식당을 선택한다. 메뉴 투표의 동점에는 무작위 방식만 제공한다.

각 Server Action은 로그인·방장·closed 상태를 확인하고, 공동 1위 옵션만 결과로 확정한다. 수동 확정은 기존처럼 모든 선택지를 선택할 수 있다. 자동 확정은 기존 decidePoll의 저장·알림 로직을 공통 함수로 호출한다.

## UI와 검증

공동 1위가 있으면 방장에게 `결정 못 하겠어요?` 영역과 사용 가능한 자동 해결 버튼을 표시한다. 순수 선택 함수에 공동 1위 제한, 거리, 방문 수, 동률 position 해소를 테스트하고, 자동 확정 action은 공동 1위 밖 옵션을 확정하지 못하게 테스트한다.
